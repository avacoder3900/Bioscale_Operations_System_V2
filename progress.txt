## Codebase Patterns

### Server File Pattern
Every `+page.server.ts` follows:
1. Import: `connectDB`, models, `requirePermission`
2. Guard: `requirePermission(locals.user, 'domain:read|write')`
3. Connect: `await connectDB()`
4. Query: Mongoose `.find()`, `.findById()`, `.aggregate()` with `.lean()`
5. Serialize: `JSON.parse(JSON.stringify(result))`
6. Return: exact shape from CONTRACT-REGISTRY.md

### Auth Pattern
- `event.locals.user` — authenticated user (set by hooks.server.ts)
- `event.locals.session` — session object
- `requirePermission(locals.user, 'permission:string')` — throws error(403) if missing
- `hasPermission(user, 'perm')` — boolean check (used in layouts for UI flags)
- `isAdmin(user)` — defined but currently unused
- Cookie: `auth-session`, SHA-256 hashed, looked up in `sessions` MongoDB collection
- Auth library: custom implementation on `@oslojs/crypto` + `@oslojs/encoding` (Lucia was removed in the Postgres→MongoDB migration)

### ID Generation
```typescript
import { generateId } from '$lib/server/db/models';
const id = generateId(); // nanoid string
```

### Denormalization Pattern
At write time, embed operator/user/project info:
```typescript
createdBy: { _id: user._id, username: user.username }
project: { _id: project._id, name: project.name, color: project.color }
```

### Sacred Document Pattern
- Check `finalizedAt` before any mutation
- Use `corrections[]` array for post-finalization fixes
- Write-once phases on CartridgeRecord: check `phase.recordedAt: { $exists: false }`

### Immutable Log Pattern
- AuditLog, ElectronicSignature, InventoryTransaction, DeviceEvent, ManufacturingMaterialTransaction
- Append only — never update or delete
- Mongoose middleware blocks `.updateOne()`, `.deleteOne()`, etc.

### API Auth Pattern (Agent API)
```typescript
// Centralized: src/lib/server/api-auth.ts (timing-safe comparison)
import { requireAgentApiKey } from '$lib/server/api-auth';
// Note: 9 endpoints still use local inline requireApiKey() — consolidation needed
```

### Deployment Rule (MANDATORY)
**NEVER deploy to Vercel from local.** Always:
1. Commit to the feature branch
2. `git push origin <branch>` to GitHub
3. Deploy from GitHub via Vercel (linked to the repo)
No `npx vercel` from local working directory. All deployments must be traceable to a GitHub commit.

## Phase 11: Mongoose Subdocument ObjectId Fix (Mar 24, 2026)

### Problem
New users created through the admin UI got 500 errors on every page load. Root cause: Mongoose auto-generates ObjectId `_id` on subdocument arrays unless explicitly disabled with `{ _id: false }`. The `user.roles[]` subdocument was missing this flag. SvelteKit cannot serialize ObjectId objects.

This only affected users whose roles were assigned through Mongoose (admin UI). Users created via raw MongoDB driver (seed scripts) were unaffected because the driver doesn't add auto-generated `_id` fields.

### Fixes applied
1. **Schema fixes** — Added `_id: false` to 11 subdocument arrays across 8 model files:
   - `user.ts`: roles, communicationPreferences
   - `bom-item.ts`: versionHistory
   - `cartridge-record.ts`: tubeRecords, parts, spectroReadings
   - `manufacturing-settings.ts`: rejectionReasonCodes
   - `opentrons-robot.ts`: recentHealthSnapshots
   - `reagent-batch-record.ts`: tubeRecords, cartridgesFilled
   - `shipping-package.ts`: cartridges
   - `test-result.ts`: readings

2. **Layout serialization** — All 8 layouts that return `locals.user` now wrap with `JSON.parse(JSON.stringify())` as defense-in-depth

3. **Data cleanup** — Removed existing ObjectId `_id` from 2 affected user accounts via `scripts/fix-role-objectids.ts`

4. **Documentation** — CLAUDE.md updated with `_id: false` pitfall

### Impact assessment
- The 11 schema fixes are preventative — most load functions already manually map fields, so the ObjectId never reached the client
- The layout serialization was the actual fix for the 500 error — layouts pass the full user object unfiltered
- The larger codebase was not affected

### Unified Timeline View (primary diagnostics view)
Stitches all 4 data sources into one chronological feed per device:
- **Firmware log lines** — extracted from `device_logs.logLines[]`, each line positioned at wall-clock time
- **Webhook round-trips** — from `webhook_logs`, expandable request/response
- **Crash reports** — from `device_crashes`, expandable checkpoint sequence trail
- **Raw events** — from `device_events`, muted styling

### Key Technical Learnings

**Timestamp computation for firmware lines:**
The firmware `ms` values are millis since boot. The Lambda's `bootTime` field is unreliable — it falls back to `new Date()` (upload time) when it can't parse a boot time from the log text. Using `bootTime + ms` causes firmware lines to appear 20+ minutes in the future.

**Correct formula:** `wallClock = uploadedAt - maxMsInChunk + line.ms`. This anchors the last line of each upload chunk at `uploadedAt` and works backwards. Firmware lines then correctly interleave with webhook/event entries that use server wall-clock time.

**Upload chunks vs sessions:**
The device uploads every ~60 seconds during idle. Each upload chunk gets metadata headers prepended (`======== UPLOAD METADATA ========`, `Boot #N | Device:...`, `Firmware: vN | Format: vN`). These are NOT separate boot sessions — multiple chunks share the same `bootCount`. The BIMS groups uploads by `bootCount` and only shows a `SESSION START` boundary when `bootCount` changes (actual reboot).

**Metadata line filtering:**
Upload metadata lines (UPLOAD METADATA, `====` separators, `Boot #N |`, `Firmware: vN |`) are filtered from the timeline and session log views. The middleware already extracts `firmwareVersion` and `bootCount` as top-level document fields, so data isn't lost.

**Clock skew between sources:**
Firmware `ms` timestamps are from the device's internal clock; webhook/event timestamps are from Lambda/Particle Cloud servers. There can be 1-3 seconds of skew. This doesn't affect debugging — the relative ordering within each source is always correct. The UI shows both the wall-clock time and the firmware `ms` offset for each entry.

1. **Oven tile hardcoded `count: 0`** in cartridge-dashboard/+page.server.ts:146-151.
2. **BackingLot.cartridgeCount never decremented** — set once at WI-01 time, stayed
   stale forever regardless of how many cartridges wax-filling pulled.
3. **Two parallel CartridgeRecord pools** that never reconciled:
   - WI-01 → nanoid-ID placeholders w/ `status='backing'` that were never consumed
   - Wax-filling `loadDeck` → UUID records from scanned barcodes (different IDs)
   Result: lineage fragmented, ghost counts inflated forever.
4. **waxQc.status='Accepted' was never written** on the happy path — only the
   reject path wrote `'Rejected'`. So "Wax QC Yield" KPI = 0 numerator.

- `cartridge-dashboard/+page.server.ts`: oven tile now queries BackingLot sum
- `wax-filling/+page.server.ts` `loadDeck`: decrements BackingLot.cartridgeCount,
  marks `consumed` at zero, voids stale WI-01 placeholders on consumption
- `wax-filling` + `opentron-control/wax/[runId]` `completeQC`: writes
  `waxQc.status='Accepted'` for every non-rejected cart
- `equipment/activity`, `equipment/location/[locationId]`, `cartridge-admin/storage`:
  status filters + BackingLot-based oven counts
- `manufacturing/consumables/+page.server.ts`: `backedCount` reads BackingLot sum

Backfills applied:
